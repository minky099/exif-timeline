# 사진 촬영시각 순차 기입 웹앱

메신저(카카오톡 등)로 받은 사진은 재압축되며 EXIF가 제거된다. iOS 사진앱은
EXIF `DateTimeOriginal`이 있으면 그 시각을 자산 촬영일로 쓰지만, 없으면
**다운로드 시각**으로 잡는다. 그래서 EXIF 없는 사진을 NAS 자동백업(Immich/Synology)하면
전부 다운로드 시각으로 몰려 순서가 뒤죽박죽이 된다.

이 웹앱은 **자동백업이 돌기 전에** 받은 사진들에 촬영시각을 순서대로 서로 다르게
EXIF로 기입한다. 아이폰 Safari에서 열어 쓴다. 서버 없이 기기 안에서만 처리한다.

- `index.html` — 본 앱 (사진 여러 장 일괄 처리)
- `exif-test.html` — 사진 실기기 최소 검증 (1장)
- `video-test.html` — 동영상(MP4) 실기기 최소 검증 (1개)

## 사용 순서 (자동백업 전에!)

> Immich/Synology 자동백업이 앨범을 훑기 **전에** 이 앱으로 처리해야 한다.
> 이미 백업된 사진은 이 앱으로 촬영일을 고쳐도 재백업 트리거가 안 될 수 있다.

1. 아이폰 Safari로 `index.html`을 연다 (호스팅 방법은 아래 참고)
2. **① 사진·동영상 선택** — 받은 JPEG/MP4 여러 개 추가 (HEIC는 자동 제외되고 안내됨)
3. **② 전체 타임라인** — 전체 시작/끝 시각을 초 단위로 입력. 이 범위가 아래 묶음 슬라이더의 좌우 끝이 된다.
4. **③ 순서 & 묶음** — ≡ 손잡이를 끌어 순서를 맞춘다(묶음 간 이동 포함). 썸네일의 **✂ 나누기**로 경계를,
   묶음 머리의 **⭱ 위와 합치기**로 경계 제거.
5. **각 묶음의 슬라이더 양 끝을 끌어** 그 묶음의 시간대를 각각 정한다. 묶음 안 사진은 그 구간에 고르게 배분된다.
6. 각 썸네일에 배정될 시각이 미리보기로 표시된다 — 확인
7. **EXIF 기입하기** → **사진 N장 한 번에 저장(공유시트)** 으로 앨범에 저장 (일괄 저장은 HTTPS 필요, 아래 참고)
8. 자동백업이 돌면 촬영시각 순서대로 정렬된다

### 아이폰에서 여는 방법
- **Docker (권장 · NAS 상시 구동):** 아래 "Docker 배포" 참고. `http://<NAS-IP>:9003` 로 접속.
- **GitHub Pages:** 저장소 Settings › Pages › Branch를 `main`으로 배포하면
  `https://<user>.github.io/exif-timeline/` 로 접속. HTTPS라 공유시트 저장까지 잘 된다.
- **로컬 파일:** `index.html`을 파일앱에 저장 후 Safari로 열어도 동작한다(오프라인 가능).
  라이브러리를 인라인해 둬서 네트워크 없이 작동한다.

## Docker 배포

단일 HTML을 nginx로 정적 서빙한다. 빌드 단계는 없다.
포트는 **9003**(HTTP) / **9443**(HTTPS). `9003`은 EXIF `DateTimeOriginal` 태그 ID `0x9003`에서 따왔다.

```bash
# (선택) 아이폰에서 https로 접속할 NAS IP를 인증서 SAN에 넣으려면:
export TLS_HOST=192.168.1.50

docker compose up -d --build
# → http://<호스트-IP>:9003   (간편 접속, 개별 저장)
# → https://<호스트-IP>:9443  (자체서명, 공유시트 일괄 저장 가능)
```
직접 빌드/실행:
```bash
docker build -t exif-timeline .
docker run -d --name exif-timeline -e TLS_HOST=192.168.1.50 \
  -p 9003:9003 -p 9443:9443 -v exif-certs:/etc/nginx/certs \
  --restart unless-stopped exif-timeline
```
업데이트(최신 코드 → 재빌드·재기동 → 안 쓰는 이미지 정리):
```bash
./update.sh
```

- 헬스체크: `GET /healthz` → `ok`
- Synology(DSM): Container Manager에서 이 저장소로 프로젝트 생성 → `docker-compose.yml` 사용.
  Immich와 같은 NAS에 함께 올려두면 편하다.

### 일괄 저장(공유시트)과 HTTPS
공유시트로 **N장을 한 번에** 앨범에 저장하는 기능(`navigator.share({files})`)은 **보안 컨텍스트에서만** 동작한다.
`http://<LAN-IP>:9003` 평문 접속은 iOS가 파일 공유 API를 막아 **롱프레스 개별 저장**만 가능하다. 두 가지 해법:

1. **자체서명 HTTPS(내장, 9443):** `https://<NAS-IP>:9443` 로 접속. 컨테이너가 시작 시
   자체서명 인증서를 만든다(`TLS_HOST`를 SAN에 포함, 인증서는 `exif-certs` 볼륨에 영속).
   아이폰에서 처음 접속하면 "신뢰할 수 없는 인증서" 경고가 뜨는데, 인증서를 한 번 신뢰하면 일괄 저장이 켜진다.
   *(iOS에서 신뢰: 인증서 프로파일 설치 후 설정 › 일반 › 정보 › 인증서 신뢰 설정에서 활성화. SAN에 접속 IP가 있어야 함.)*
   `TLS_HOST`를 바꿔 재발급하려면 `docker volume rm exif-certs` 후 재기동.
2. **리버스 프록시 + 정식 인증서(권장):** Synology 리버스 프록시로 `https://` 도메인(예: `*.synology.me`,
   Let's Encrypt)을 9003 앞에 붙이면, 경고 없이 신뢰된 HTTPS로 일괄 저장이 바로 된다.

## 저장 방식 (iOS)

우선순위대로 시도한다. **실제 이미지 파일을 그대로 넘기는 방식**이라야 EXIF가 보존된다.

1. **공유시트 일괄 저장 (기본, HTTPS 필요):** "사진 N장 한 번에 저장" → 시트에서 **이미지 저장** 선택.
   N장을 **한 번에** 앨범에 넣는다. (`navigator.share({files})` 사용, 보안 컨텍스트=HTTPS/localhost에서만 활성)
2. **미리보기 롱프레스 (개별):** 결과 이미지를 **길게 눌러 "사진에 추가"**. HTTP 접속 시 이 방법만 가능.
3. **파일앱 경유:** 각 이미지 다운로드 → 파일앱 → 사진앱으로 가져오기.

> 일괄 저장이 안 보이면 HTTP로 접속 중이라는 뜻이다. 위 **Docker 배포 › 일괄 저장과 HTTPS** 참고.

### 가정이 깨질 때의 대안
저장한 사진의 촬영일이 여전히 저장 시각으로 잡힌다면:
- **저장 방식을 바꿔본다** (위 1 → 2 → 3 순서로). 기기/iOS 버전에 따라 EXIF를 읽는 경로가 다르다.
- **최후 수단 — 메타데이터 편집 앱 경유:** 이 앱으로 EXIF를 넣은 JPEG를 저장한 뒤,
  [Metapho](https://apps.apple.com/app/metapho/id914457352) 같은 앱으로 열어
  촬영일을 확정 적용한다. Metapho는 사진앱 자산의 날짜를 직접 고칠 수 있어 가장 확실한 대비책이다.

## 시간 분배 로직 (그룹별 시간창)

전체 타임라인 `[S, E]` 위에서 사진을 **묶음(그룹)** 으로 지정하고, **각 묶음의 시간창을 슬라이더로 직접** 정한다.
(예전의 단일 "경계 가중치" 방식 대신, 묶음마다 시작·끝을 독립적으로 지정.)

- 전체 시작 `S`, 끝 `E`. 각 묶음 `g`는 `[S, E]` 위의 자기 구간 `[gs_g, ge_g]`(듀얼 슬라이더로 조절)을 가진다.
- 묶음 슬라이더는 서로 **교차하지 않게** 이웃과 맞물려 클램프된다(묶음 순서 유지).
- 묶음 안 `k_g`장은 그 구간에 **고르게 배분**: `j`번째 = `gs + (ge − gs)·j/(k_g − 1)` (`j = 0 … k_g−1`).
  - `k_g == 1`이면 구간 중앙에 배치.
  - 구간 폭이 너무 좁으면 묶음 안에서 최소 1초 간격을 보장.
- 마지막으로 **전 사진이 순서대로 서로 다른 시각**이 되도록 전역 단조 증가를 보정한다
  (같거나 역전되면 앞 사진 +1초).
- 기본값: 묶음이 생기면 `[S, E]`를 묶음 수만큼 **균등 분할**해 초기 구간을 잡는다. 이후 슬라이더로 조정.
  (묶음 **수**가 바뀌면 균등 분할로 초기화, 순서만 바뀌면 조정값 유지.)

**예:** 전체 14:00:00 - 16:00:00, 2묶음 `[3, 2]`장.
묶음1 슬라이더를 `14:00 - 14:24`, 묶음2를 `15:24 - 16:00`로 맞추면:
```
[묶음1] 14:00:00  14:12:00  14:24:00     [묶음2] 15:24:00  16:00:00
```

## 동영상(MP4) 지원 — 실기기 검증 완료, 본 앱 통합됨

카톡으로 받는 동영상은 MP4다. 동영상은 EXIF가 아니라 **MP4 컨테이너의
`mvhd`/`tkhd`/`mdhd` 박스에 있는 `creation_time`**(1904-01-01 UTC 기점 초)이 촬영시각 역할을 한다.
**실기기 검증 결과: iOS 사진앱이 이 값을 촬영일로 인식한다** (2026-07 확인).

본 앱(index.html)에서 사진(JPEG)과 동영상(MP4)을 **같은 타임라인·같은 묶음에 섞어** 올릴 수 있다.
동영상은 첫 프레임 썸네일과 `▶ MP4` 배지로 표시되고, 기입 시 컨테이너 시각을
**재인코딩 없이 바이너리 제자리 패치**한다(v0 32비트/v1 64비트, `modification_time` 포함, 파일 크기 불변).

저장 시 유의:
- **HTTPS 접속(권장):** 공유시트 일괄 저장이 사진·동영상을 한 번에 앨범에 넣는다.
- **HTTP 접속:** 사진은 롱프레스 "사진에 추가"가 되지만 **동영상에는 롱프레스 저장이 없다** —
  다운로드 → 파일앱 → 공유 → 비디오 저장 경로만 가능하므로, 동영상을 쓸 거면 HTTPS 접속을 권한다.

참고: 아이폰 원본 영상에는 애플 QuickTime 날짜 키(`com.apple.quicktime.creationdate`)가 있어
iOS가 그 값을 우선 읽을 수 있다. `video-test.html`(단일 검증 페이지)이 이 키를 감지해 경고한다
(카톡 재인코딩본에는 보통 없음).

## 제약 / 아키텍처

- **단일 HTML 파일, 클라이언트 전용.** 서버·빌드 없음. 사진은 기기 밖으로 나가지 않는다.
- **JPEG 전용.** EXIF 기입에 [piexifjs](https://github.com/hMatoba/piexifjs) 1.0.6 사용.
  piexifjs는 HEIC/HEIF를 지원하지 않으므로 HEIC 입력은 매직바이트로 감지해 제외하고 안내한다.
- 기입 필드: `DateTimeOriginal`(0x9003) + `DateTimeDigitized`(0x9004) + 0th `DateTime`(0x0132).
  기존 EXIF(방향 태그 등)는 최대한 보존하며, 재기입 실패 시 촬영시각+방향만 남기는 최소 구성으로 대체한다.
- 라이브러리는 각 HTML에 **인라인**되어 있다. CDN/네트워크 불필요, 오프라인 동작.
  로드 실패 시 침묵하지 않고 명확한 경고를 띄운다.

### piexifjs 인라인 재빌드
`index.html` / `exif-test.html`의 첫 번째 `<script>` 블록이 piexifjs 1.0.6(`piexif.js`, `piexif.min.js` 아님)의
전체 소스다. 갱신하려면:
```bash
npm pack piexifjs@1.0.6           # piexifjs-1.0.6.tgz → package/piexif.js
# 각 HTML의 첫 <script>…</script> 안 내용을 package/piexif.js 로 교체
```

### 표기 규칙
- 숫자 범위는 물결(`~`)이 아니라 하이픈(`-`)으로 쓴다. 예: `14:00 - 14:30`.
